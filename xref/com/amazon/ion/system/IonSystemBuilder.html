28  /**
29   * The builder for creating {@link IonSystem}s.
30   * Most applications will only have one or two system instances;
31   * see {@link IonSystem} for important constraints.
32   * <p>
33   * Builders may be configured once and reused to construct multiple
34   * objects. They can be {@link #copy() copied} to create a mutable
35   * copy of a prototype (presumably for altering some property).
36   * <p>
37   * <b>Instances of this class are not safe for use by multiple threads unless
38   * they are {@linkplain #immutable() immutable}.</b>
39   * <p>
40   * The easiest way to get going is to use the {@link #standard()} builder:
41   *<pre>
42   *    IonSystem ion = IonSystemBuilder.standard().build();
43   *</pre>
44   * <p>
45   * However, most long-lived applications will want to provide a custom
46   * {@link IonCatalog} implementation rather than using the default
47   * {@link SimpleCatalog}.  For example:
48   *<pre>
49   *    IonCatalog catalog = newCustomCatalog();
50   *    IonSystemBuilder b = IonSystemBuilder.standard().copy();
51   *    b.setCatalog(catalog);
52   *    IonSystem ion = b.build();
53   *</pre>
54   * <p>
55   * Configuration properties follow the standard JavaBeans idiom in order to be
56   * friendly to dependency injection systems.  They also provide alternative
57   * mutation methods that enable a more fluid style:
58   *<pre>
59   *    IonCatalog catalog = newCustomCatalog();
60   *    IonSystem ion = IonSystemBuilder.standard()
61   *                                    .withCatalog(catalog)
62   *                                    .build();
63   *</pre>
64   *
65   * <h2>Configuration Properties</h2>
66   * <p>
67   * This builder provides the following configurable properties:
68   * <ul>
69   *   <li>
70   *     <b>catalog</b>: The {@link IonCatalog} used as a default when reading Ion
71   *     data. If null, each system will be built with a new
72   *     {@link SimpleCatalog}.
73   *   </li>
74   *   <li>
75   *     <b>streamCopyOptimized</b>: When true, this enables optimizations when
76   *     copying data between two Ion streams. For example, in some cases raw
77   *     binary-encoded Ion can be copied directly from the input to the output.
78   *     This can have significant performance benefits when the appropriate
79   *     conditions are met. <b>This feature is experimental! Please test
80   *     thoroughly and report any issues.</b>
81   *   </li>
82   * </ul>
83   */

331     /**
332      * Builds a new {@link IonSystem} instance based on this builder's
333      * configuration properties.
334      */
335     public final IonSystem build()
336     {
337         IonCatalog catalog =
338             (myCatalog != null ? myCatalog : new SimpleCatalog());
339 
340         IonTextWriterBuilder twb =
341             IonTextWriterBuilder.standard().withCharsetAscii();
342         twb.setCatalog(catalog);
343 
344         _Private_IonBinaryWriterBuilder bwb =
345             _Private_IonBinaryWriterBuilder.standard();
346         bwb.setCatalog(catalog);
347         bwb.setStreamCopyOptimized(myStreamCopyOptimized);
348 
349         // TODO Would be nice to remove this since it's implied by the BWB.
350         //      However that currently causes problems in the IonSystem
351         //      constructors (which get a null initialSymtab).
352         SymbolTable systemSymtab = _Private_Utils.systemSymtab(1);
353         bwb.setInitialSymbolTable(systemSymtab);
354         // This is what we need, more or less.
355 //        bwb = bwb.fillDefaults();
356         IonReaderBuilder rb = readerBuilder == null ? IonReaderBuilder.standard() : readerBuilder;
357         rb = rb.withCatalog(catalog);
358         return newLiteSystem(twb, bwb, rb);
359     }
